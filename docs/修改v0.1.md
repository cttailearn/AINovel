# 二、后端问题（按严重度排序）

## 🔴 P0 — 影响主流程 / 数据一致性

### 1. database.py 单文件 3,739 行（schema + 89 个 CRUD + 迁移全堆一起）

症状 ：维护噩梦，定位一个函数要翻 100+ 行；新人上手成本极高；schema 变更风险大。 建议 ：拆成

这样 review、迁移测试、grep 都能聚焦到一个领域。

### 2. 硬编码 API_HOST/API_PORT = 127.0.0.1:8008

位置 ： backend/config.py:12-13 症状 ：注释里写"不要改"，但 CI/测试/多实例场景需要切端口；当前用 _preflight_check_port + SO_REUSEADDR 兜底，本质是「不让你用错」。 建议 ：

- 保留默认 8008，但允许 AINOVEL_PORT 环境变量覆盖
- 端口冲突时不要直接 sys.exit(1) ，而是把端口写到一个 data/.port 文件，前端启动时读这个文件决定访问哪个端口（顺便解决 Tauri 端口随机化场景）

### 3. 两个 KG 系统并行存在，ai_kg_*（项目级，11 张表）vs characters/events/...（小说级，5 张表）

症状 ： docs/ai-creation-kg-architecture.md 自己承认「物理隔离」，但运行时谁也用不上谁——小说级 KG 在创作工坊里完全被忽略，反之亦然。 建议 ：定义 kg_system enum 字段，把两套表合成视图 v_project_kg / v_novel_kg ；或者干脆把「小说级 KG」标记 deprecated，所有新功能只走 ai_kg_* 。

### 4. create_project / update_project 自动 seed KG 时只打 logger.warning，UI 完全无感

位置 ： services/creation_service.py:148-150 和 184-194 症状 ：用户改了 initial_concepts 后 KG 没更新，但前端没有任何反馈，会让用户以为是 LLM 抽取得不好。 建议 ：返回结构改成 {ok, seed_warning?, kg_stats} ，让前端能提示「已重新灌入，但 LLM 后抽取的 X 个实体保留了」。

### 5. _run_migrations 是单条 ALTER 串行执行，对长链路项目无原子保证

位置 ： database.py:473-672 症状 ：跑到一半断电会留下半迁移状态（SQLite 没有 DDL 事务包裹所有 ALTER）。 建议 ：用 user_version PRAGMA 记录迁移进度；新增一个迁移幂等检查器，迁移开始前 set user_version = old + 1 ，失败时回滚到 old 。

## 🟠 P1 — 影响性能 / 可维护性

### 6. 每个请求 get_db() 都开/关一个新连接，无连接池

位置 ： database.py:675-682 症状 ：在创作三 Agent 流水线里（每次 critic→planner→writer 都要查 KG、写章节），每个子操作都开连接，开销 + SQLite 文件锁竞争。 建议 ：用 aiosqlite.connect 单连接 + check_same_thread=False ，或者用 FastAPI Depends 把连接注入到请求作用域。

### 7. 所有 KG CRUD 端点都先 await creation_service.get_project_detail(project_id)

位置 ： routers/creation.py 的几乎每个 KG 端点 症状 ：每次手动新增/编辑一个 KG 节点，都要拉一次完整的 project + chapters + kg_stats + locations + plot_threads + themes + kg_full 全量数据。 建议 ：拆出一个轻量 _assert_project_exists(project_id) -> None ，只查 id 字段。

### 8. _resolve_model_cfg 在多个 service 里被复制（creation_service / creation_intake_service）

症状 ：逻辑漂移风险——一个改了 enabled=1 检查，另一个没有。 建议 ：提到 services/ai_service.py 作为 resolve_chat_model(model_id) 公共函数。

### 9. services/ai_service.py 只有 280 行，但承载了 4 个 provider 的实现

症状 ：加一个 provider 要在这个文件里同时改路由、provider 类、test 连接逻辑。 建议 ：抽 providers/{anthropic,openai,custom}.py ，每个 provider 自己实现 chat/embed/test。

### 10. paragraphDiff.py（后端）+ textDiffUtil.js（前端）算法重复

症状 ：改一处 diff 行为（比如改 LCS 阈值），前端后端必须同步。 建议 ：后端只返回 {added, removed, unchanged} segments 列表，前端只做渲染；或后端用 SSE 把 diff 一次性推过来。

### 11. 创作流程 generate_chapter_streaming 在生成前会先 delete_ai_chapter 同 chapter_no 的旧章节

位置 ： services/creation_service.py:605-608 症状 ：用户点「重新生成」会清空之前的 variant 与 KG 抽取，UX 上是「回退」，但前端没二次确认。 建议 ：加 force=true query 参数 + 默认弹确认对话框；或把旧 chapter 标记 superseded 而不是 DELETE。

### 12. await db.update_ai_chapter(int(ch["id"])) 漏掉 kg_extracted 状态重置

症状 ：被删除重建的章节在 ai_chapters 里 kg_extracted=0 ，但被关联的实体 ID 仍在 ai_kg_characters.source_chapter_id 指向老 ID（已不存在），变成「孤儿证据」。 建议 ：删除章节时同时清掉对应 source_chapter_id （用 ON DELETE SET NULL ，已有 FK 应该 OK，但要看现有数据有没有孤儿）。

## 🟡 P2 — 体验 / 安全 / 一致性

### 13. 错误脱敏 _SENSITIVE_PATTERNS 只覆盖 OpenAI/Anthropic 风格 key

位置 ： services/creation_service.py:52-59 建议 ：扩展正则，覆盖 sk- / gsk- （Groq）、 xoxb- （Slack 风格）、国产 sk-xxx 、 API_KEY= 自定义格式；同时把 192.168/10.x 之外的 172.16-31.x 也补上。

### 14. 多个 SSE 端点用 HTTPException 抛错，但客户端是 EventStream，HTTP 错误会被吞掉

位置 ： routers/creation.py:619 、 635 、 routers/enrichment.py 症状 ：前端 XHR 拿到的是 status=409 但 body 是普通文本，error toast 文案会很奇怪。 建议 ：把所有 SSE 入口的错误先 await record.publish({"event": "error", "message": ...}) 再 yield，HTTP 层只负责 200。

### 15. 章节序号在 delete_ai_chapter 后「回退」，但 project.current_chapter_no 没保证不出现空洞

症状 ：用户删了第 5 章，重新生成时 chapter_no 不会自动补 5 而是按 current_chapter_no 走。 建议 ：加一个 next_available_chapter_no(project_id) ，每次生成前查询可用空号。

### 16. kg_extracted 字段只是 0/1 标记

位置 ：架构文档 §1.1 已自认问题 建议 ：见架构文档阶段 1 的①，加 kg_extracted_at + kg_entity_count + kg_event_count （migration 已加了），但前端没消费这个时间戳。

### 17. 富文本/章节内容没有字数上限

位置 ： AiChapterContentUpdate.content: min_length=0 ，无 max 症状 ：用户粘贴 100MB 文本不会被拒。 建议 ：加 max_length=500_000 （约 50 万字符 ≈ 30 万汉字），保护 SQLite。

### 18. image_service.py 把 base64 / 远程 URL 都让前端处理

症状 ：base64 数据超大时存 localStorage 容易爆。 建议 ：后端把 base64 转文件存到 data/image_cache/ ，返回 /api/image/cache/xxx URL，前端只引用 URL。

### 19. 没有限流：用户连续点 10 次「生成第 N 章」，后端会真的并发跑 10 个 LLM 任务

建议 ： registry.register 已经做了 (kind, subject_id) 互斥，但同 project 跨章节没有——加 (kind, project_id) 维度的互斥。

# 三、前端问题

## 🔴 P0

### 20. App.jsx 把 NAV_ITEMS / PAGE_META 写死，路由也只是 useState

位置 ： frontend/src/App.jsx:65-121, 142 症状 ：浏览器刷新会丢失当前页面、无法分享深链、无法前进后退。 建议 ：用 react-router 6 + `<NavLink>` ，把 ?page=creation&project=42 写到 URL。

### 21. api/client.js 711 行 + 全部 API 集中在一处

症状 ：tree-shaking 失效，新增 endpoint 必须改这个文件，PR 冲突率高。 建议 ：

### 22. 多种持久化并存：localStorage × 后端 mirror × 内存 state

位置 ： CreationTaskContext.jsx 同时写 localStorage 和 PUT /tasks/mirror 症状 ：localStorage 写入是同步的，每次状态变更都触发；mirror API 失败时静默吞；多 tab 同时打开时 localStorage 会被互相覆盖。 建议 ：

- 用 BroadcastChannel 同步多 tab
- 合并持久化层，把 state 序列化抽到 useTaskPersist(key, state) hook
- mirror API 失败加 retry + UI 提示「任务未同步到云端，刷新可能丢失」

### 23. 关键状态变量散落在 5 个 Context 里，组件订阅粒度太粗

症状 ： useCreationTask() 返回 11 个字段，任何一个变化都让所有消费者 re-render。 建议 ：用 selector 模式（zustand）或拆成多个小 Context（ useTaskProgress() / useTaskControl() ）。

## 🟠 P1

### 24. 大列表没有虚拟滚动

位置 ： Workbench.jsx 、 ChapterList.jsx 、 KnowledgeGraphPanel.jsx 、 KGGraphView.jsx 症状 ：100+ 章节 / 500+ KG 节点时渲染卡顿。 建议 ：用 react-window 或 @tanstack/react-virtual ，KG 视图用 cytoscape.js / vis-network 。

### 25. ChapterGenerator.jsx 的 STAGE_LABELS 引用 writer_2_done/critic_2_done，但当前流水线是单候选

位置 ： components/creation/ChapterGenerator.jsx:11-15 症状 ：这些 stage 永远不会被触发，但代码还在 — 误导新人以为有 3 候选。 建议 ：从 STAGE_LABELS 删掉 writer_2_done / critic_2_done，把 SUB_STAGE_PROGRESS 也同步收敛。

### 26. VariantCards.jsx 命名与实际 UI 不一致

症状 ：组件名是 Cards 但实际渲染的是 Tabs。 建议 ：改名 VariantTabs （内部已有别名）或统一改回 Card。

### 27. 错误边界只有一个，包在 App 外层

症状 ：Workbench 抛错时整个白屏；Creation 抛错时整个 studio 白屏。 建议 ：每个一级页面（Workbench / CreationStudio / ImageGenerationPage / Settings）各包一个 ErrorBoundary + 「仅此面板重试」按钮。

### 28. getOrCreateClientId() 生成规则 cli_${Date.now()}_$ 可能跟其他 ainovel.* 的 v1 key 冲突

建议 ：抽到 utils/clientId.js 用 crypto.randomUUID() 。

### 29. postStream 用 xhr.responseText.slice(buffer.length) 增量解析 SSE

症状 ： readyState === 3 时可能拿到不完整的 chunk（XHR 在流式下不一定保证边界）。 建议 ：用 fetch + ReadableStream （已经在 getEventStream 里实现），统一改造成 api.streamPost(path, body, {onEvent}) 一套。

### 30. 图像生成页面结果只存 localStorage 30 条，没有磁盘持久化

建议 ：生成结果超过 30 条时自动落盘到后端 /api/image/cache ，URL 引用即可。

### 31. Workbench 和 CreationStudio 是两个独立一级页面，工作流跨不过去

症状 ：用户场景「上传一本《斗破苍穹》前 30 章做样本 → 让 AI 续写第 31 章」无法直接做。 建议 ：在 CreationStudio 里加「导入小说章节作为前情提要」入口，把 novel KG 投影到 project KG。

## 🟡 P2

### 32. topSearch 全局搜索框但只在 workbench 和 image 实际生效
