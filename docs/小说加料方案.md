# 小说加料 (AI Enrichment) 功能方案

> **版本**：v0.2 实施版
> **状态**：已实现
> **日期**：2026-06-08
> **变更对比**：v0.2 在 v0.1 基础上新增「拆分/总结专属子页 + AI 改写迁入阅读器 + enrichment_suggestions 应用历史表」

---

## 1. 背景与目标

### 1.1 现状
项目当前 (`AINovel`) 是一套本地运行的小说管理工具，主要能力：
- 上传 TXT 小说、解析章节、原生窗口阅读（`Workbench` + `NovelReader`）。
- 知识图谱抽取（5 阶段 LLM 流水线，写入 `characters / events / *_relations` 五张表）。
- 提示词管理（`prompt_templates` 表 + `PromptManagerPanel`），已支持 `kg.*` 分类与「系统破甲 / 场景识别 / 改写规则」等 Tab。
- 模型配置（`model_configs` 表 + `ModelConfigPanel`）。

### 1.2 目标
新增 **小说加料 (AI Enrichment)** 模块，对应截图中的 5 步流水线：
1. 书籍拆分（章节解析，已具备）
2. 内容总结（AI 给每章写一段摘要）
3. 识别待处理（AI 抽取登场人物 / 关键事件 / 场景分类）
4. AI 改写（按场景 + 改写规则对章节正文加料）
5. 合并输出（按顺序拼回为一份加料版 TXT）

加料结果落库到独立表 `chapter_enrichments`，与 `chapters` 一对一，便于失败重跑、原地编辑、最终导出。

**v0.2 关键升级**：
1. 拆分 / 总结 / 加料三个子视图以 **Tabs 形式保留在加料工作台内**。
2. AI 改写入口 **从工作台迁入阅读器**（`NovelReader` 右侧 AI 加料面板），点"应用"一键写回原文。
3. 新增 `enrichment_suggestions` 表记录每次"应用"，保留历史 + 支持回滚。
4. 导出 TXT 改为基于 `chapters.content`（已合并的版本）而非 `rewrite_text`。

### 1.3 与现有模块的关系
| 现有能力 | 复用方式 |
|---|---|
| `chapters` 表（章节正文 + 起止位置） | 继续作为加料的输入源，不破坏既有解析结果 |
| `prompt_templates` 表 + `PromptManagerPanel` | 在 `PROMPT_CATEGORIES` 中新增 `enrichment` 分类，UI 增加「小说加料」Tab，复用通用编辑/启停 |
| `model_configs` 表 | 复用为加料任务的 chat 模型选择入口 |
| `model_service` / `ai_service.chat_completion` | 复用底层 LLM 调用 |
| `step-bar` / `ExtractionProgress` / `kg-progress` 样式 | 复用到新模块的 5 步进度展示 |
| `workbench-shell` / `library-shell` 布局 | 复用到新页面外壳 |

## 2. 数据模型

### 2.1 新增表 `chapter_enrichments`
每条记录对一本书的一个章节保存一次「加料结果」。`status` 字段驱动重试与汇总。

```sql
CREATE TABLE chapter_enrichments (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  novel_id INTEGER NOT NULL,
  chapter_id INTEGER NOT NULL,
  -- 五个步骤的产物，单独落库便于单步重跑
  summary TEXT,                          -- 内容总结
  summary_status TEXT DEFAULT 'pending', -- pending / running / done / failed / skipped
  summary_error TEXT,
  summary_model_id INTEGER,

  recognition_json TEXT,                 -- 登场人物 + 关键事件 + 场景标签 (JSON)
  recognition_status TEXT DEFAULT 'pending',
  recognition_error TEXT,
  recognition_model_id INTEGER,

  rewrite_text TEXT,                     -- 加料后正文
  rewrite_status TEXT DEFAULT 'pending',
  rewrite_error TEXT,
  rewrite_model_id INTEGER,

  -- 公共字段
  scene_tag TEXT,                        -- 复用识别阶段的场景标签 (e.g. "高燃战斗场景")
  status TEXT NOT NULL DEFAULT 'pending',-- 汇总状态: pending / running / done / failed / partial
  error TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (novel_id) REFERENCES novels(id) ON DELETE CASCADE,
  FOREIGN KEY (chapter_id) REFERENCES chapters(id) ON DELETE CASCADE,
  UNIQUE (chapter_id)
);
CREATE INDEX idx_enrichments_novel ON chapter_enrichments(novel_id);
CREATE INDEX idx_enrichments_status ON chapter_enrichments(status);
```

> 「识别结果」用 JSON 存储，因为 `登场人物 / 关键事件 / 场景标签` 三个结构来自同一个 LLM 调用，没必要拆三张表；查询时只在章节详情里展开。

### 2.2 迁移
在 `database.py` 的 `_run_migrations` 末尾追加建表语句，使用 `CREATE TABLE IF NOT EXISTS`，无需手动迁移。

### 2.3 不复用 `characters / events` 表的考虑
知识图谱是**全本**维度（人物会去重 + 关系合并），而加料是**单章**维度（同一人物每章都会被重新识别一次，且要保留「本章内表现」）。强行写 `characters` 表反而要加新列、破坏既有语义。**保持独立**。

## 3. 后端 API

新增 `backend/routers/enrichment.py`，全部走 `/api/enrichment` 前缀。

### 3.1 端点一览

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/enrichment/novels/{novel_id}/summary` | 列出该书每章的加料进度（章节 + summary/recognition/rewrite 三态 + 汇总），用于右侧「总结统计」与左侧章节导航 |
| GET | `/api/enrichment/chapters/{chapter_id}` | 取单个章节的加料详情（summary / recognition_json / rewrite_text / 错误信息） |
| POST | `/api/enrichment/chapters/{chapter_id}/summary` | 触发**单章**「内容总结」 |
| POST | `/api/enrichment/chapters/{chapter_id}/recognition` | 触发**单章**「识别」 |
| POST | `/api/enrichment/chapters/{chapter_id}/rewrite` | 触发**单章**「改写」 |
| POST | `/api/enrichment/novels/{novel_id}/batch` | 批量触发整本加料（可指定仅跑哪几步），支持 SSE 流式返回进度 |
| PUT | `/api/enrichment/chapters/{chapter_id}` | 手动编辑某个字段（如手动改 summary / rewrite_text） |
| GET | `/api/enrichment/novels/{novel_id}/export` | 把所有 `rewrite_status=done` 的章节拼成一份 TXT，下载 |
| POST | `/api/enrichment/novels/{novel_id}/reset` | 清空该书所有加料记录（仅清 enrichment 表，章节/原书保留） |

### 3.2 请求 / 响应形态

#### POST /api/enrichment/chapters/{chapter_id}/summary
请求体：
```json
{
  "model_config_id": 1,
  "prompt_key": "enrichment.summary",
  "override_prompt": null
}
```
响应：`{ "status": "done", "summary": "...", "model_id": 1, "duration_ms": 1820 }`

#### POST /api/enrichment/novels/{novel_id}/batch
请求体：
```json
{
  "model_config_id": 1,
  "steps": ["summary", "recognition", "rewrite"],
  "chapter_ids": [12, 13, 14],
  "concurrency": 2,
  "skip_existing": true
}
```
SSE 事件流（`event: progress`）：
```json
{
  "step": "summary",
  "done": 3,
  "total": 10,
  "current_chapter": 4,
  "current_title": "第 4 章 ..."
}
```
结束后发 `event: complete` 携带最终统计。

#### GET /api/enrichment/novels/{novel_id}/export
- 服务端把 `chapters` 按 `chapter_number` 排序，串接 `rewrite_text`（无 rewrite 则退回 `content`）。
- Header: `text/plain; charset=utf-8`，filename=`{novel.title}.enriched.txt`。
- 注意：**写一个新文件**，不覆盖原书（避免破坏知识图谱训练数据）。

### 3.3 错误处理
- LLM 调用失败 → 落 `*_status=failed` + `*_error` 字段，前端红色徽标。
- 解析 JSON 失败 → 同上，错误信息保留原始 LLM 输出（前 500 字）便于排查。
- 整批失败 → SSE 推 `event: error` 携带 `code` + `message`。

## 4. 提示词设计

### 4.1 新增分类 `enrichment`
在 `backend/services/prompt_service.py` 的 `PROMPT_CATEGORIES` 追加：
```python
{
    "key": "enrichment",
    "label": "小说加料",
    "description": "章节摘要、登场人物/关键事件识别、场景分类与改写规则的提示词。"
}
```

### 4.2 新增 4 个内置模板（在 `DEFAULT_PROMPTS` 中追加）

| key | 用途 | 关键变量 |
|---|---|---|
| `enrichment.summary` | 单章 100~200 字摘要 | `{chapter_title}`, `{chapter_text}` |
| `enrichment.recognition` | 抽取人物/事件/场景标签 | `{chapter_title}`, `{chapter_text}`, `{summary}` |
| `enrichment.scene_classify` | 场景识别（独立 prompt，便于按场景切换改写规则） | `{chapter_title}`, `{chapter_text}` |
| `enrichment.rewrite` | 加料改写正文 | `{chapter_title}`, `{chapter_text}`, `{summary}`, `{recognition_json}`, `{scene_tag}`, `{scene_rule}`, `{general_rule}` |

#### 4.2.1 复用现有「场景识别 / 改写规则」Tab
截图 2 中的「场景识别」「改写规则」**就是** `PromptManagerPanel` 已有的 Tab 入口。本次新增的「小说加料」Tab 默认会列出以上 4 个新 prompt，但用户也可以在「场景识别」「改写规则」分类下创建多份场景特定规则。前端从这两个分类下拉选择，传入后端 `enrichment.rewrite` 的 `override_prompt` 字段。

#### 4.2.2 改写规则的多层合并
服务端组装改写 prompt 时按以下顺序拼接：
1. 通用指导（`enrichment.rewrite` 的 system_prompt）
2. 用户在「改写规则」分类下选中的「通用规则」子模板内容
3. 识别出的 `scene_tag` 命中的「场景特定」子模板内容
4. 章节级变量（标题 / 摘要 / 人物事件 JSON）

这样既支持「统一改写」也支持「按场景差异化」。

## 5. 后端服务

新增 `backend/services/enrichment_service.py`，主要函数：

| 函数 | 职责 |
|---|---|
| `list_novel_progress(novel_id)` | JOIN `chapters` + `chapter_enrichments`，返回每章三态 |
| `get_chapter_enrichment(chapter_id)` | 取详情 |
| `run_step_summary(chapter_id, model_cfg, prompt_key, override)` | 调 LLM → 写 `summary` + 状态 |
| `run_step_recognition(chapter_id, model_cfg, prompt_key, override)` | 调 LLM → 解析 JSON → 写 `recognition_json` + `scene_tag` |
| `run_step_rewrite(chapter_id, model_cfg, prompt_key, override)` | 拉 summary + recognition + scene_rule → 拼接 prompt → 调 LLM → 写 `rewrite_text` |
| `run_batch(novel_id, model_cfg, steps, chapter_ids, concurrency, skip_existing, on_event)` | 异步并发调度；用 `asyncio.Semaphore` 控制并发；`on_event` 推送 SSE |
| `export_enriched_txt(novel_id)` | 串接并落盘到 `data/exports/` |
| `reset_novel(novel_id)` | DELETE FROM chapter_enrichments WHERE novel_id=? |

并发策略：参考 `agents.py` 的 `MAX_CONCURRENCY` 默认 2；过高的并发容易被 LLM 端限流并把失败率拉高。

## 7. 端到端工作流

1. 用户在「加料工作台」选一本已解析章节的小说（或新建并复用 Workbench 的上传/解析流程）。
2. 选模型 → 点「开始处理」。后端 SSE 实时推送：
   - summary: 3/10、4/10...
   - recognition: 0/10、1/10...
   - rewrite: 0/10、1/10...
3. 任一步骤完成后，前端拉一次 `listProgress` 刷新徽标；点开具体章节可看「情节概要 / 登场人物 / 关键事件 / 改写正文」。
4. 失败章节可单章重跑（卡片右上「重新生成」按钮）或「重试失败章节」批量。
5. 5 步全完成后点「合并输出」→ 浏览器下载 `{title}.enriched.txt`。

## 9. 风险与权衡

| 项 | 风险 | 应对 |
|---|---|---|
| LLM 成本 | 长篇小说每章多次 LLM 调用很贵 | Step 1/2/3 默认 `skip_existing=true`；用户可单章重跑 |
| 改写质量 | 通用 prompt 在不同题材上表现差异大 | 通过场景分类 + 用户可改 prompt，把调优权交给用户 |
| 并发限流 | LLM 端可能 429 | `MAX_CONCURRENCY=2`；`retries=2` 已存在；429 在前端降速提示 |
| 数据膨胀 | `recognition_json` 累积较大 | 一次保存就一张 JSON，单章 1~5KB；可接受 |
| 用户体验 | 整本跑完再给反馈会焦虑 | 用 SSE 实时刷新；可中断；失败可单章重试 |

## 11. 验收用例

1. 上传 TXT → 解析章节 → 进入「加料工作台」→ 选模型 → 点「开始处理」→ 5 步 StepBar 进度动起来。
2. 处理过程中可中断，状态持久化；刷新页面后徽标正确。
3. 单章 summary / recognition / rewrite 任一失败时，红色徽标 + 「重新生成」可点；点击后单步重跑成功。
4. `recognition_json` 中的人物和事件在 `ChapterDetail` 卡片里正确渲染。
5. 5 步全完成后点「导出 TXT」，下载文件按 `chapter_number` 顺序拼接，与 `NovelReader` 看到的章节顺序一致。
6. 在「提示词管理」修改 `enrichment.rewrite`，后续 `run_rewrite` 立即使用新 prompt。
7. 在「改写规则」分类下创建新模板并指定为「通用规则」，`run_rewrite` 自动拼接该内容。

## 12. v0.2 增量变更

### 12.1 新增数据表 `enrichment_suggestions`

> 每次"应用"留一条记录，支持回滚与历史查看。

```sql
CREATE TABLE IF NOT EXISTS enrichment_suggestions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    chapter_id INTEGER NOT NULL,
    novel_id INTEGER NOT NULL,
    enrichment_id INTEGER,
    original_snapshot TEXT NOT NULL,
    rewrite_text TEXT NOT NULL,
    model_id INTEGER,
    summary_snapshot TEXT,
    recognition_snapshot TEXT,
    scene_tag TEXT,
    status TEXT NOT NULL DEFAULT 'applied',  -- applied / superseded / reverted
    applied_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    reverted_at TIMESTAMP,
    FOREIGN KEY (chapter_id) REFERENCES chapters(id) ON DELETE CASCADE,
    FOREIGN KEY (novel_id) REFERENCES novels(id) ON DELETE CASCADE,
    FOREIGN KEY (enrichment_id) REFERENCES chapter_enrichments(id) ON DELETE SET NULL
);
```

**业务规则**：
- 当前 `applied` 被新"应用"覆盖时，旧记录 `status` 自动改 `superseded`，原文中 `chapters.content` 写入新内容。
- 「回滚」把目标 `superseded` 改回 `applied`，再把当前 `applied` 改 `reverted`，正文变回目标 `rewrite_text`。
- 「清空加料」对每章最早的 `original_snapshot` 还原 `chapters.content`，再删两张表。

### 12.7 验收用例 (v0.2)

1. 加料工作台顶部出现 3 个 Tabs，可来回切换不丢失状态。
2. "拆分" 页表格化展示所有章节 + 起止 + 字数，可搜索。
3. "总结" 页表格 + 状态点 + 场景标签 + 摘要预览 + 详情抽屉；点"跑总结"用选定模型批量跑。
4. 阅读器内点 "AI 加料" 打开右侧面板，能看到三步状态。
5. 一键生成后，章节有 `rewrite_text` 时，"预览高亮" 把原文 + 改写 diff 渲染在主内容区。
6. 点 "应用到原文"，章节正文变长，弹 Toast "已应用: 净增 XXX 字"，历史弹窗可看到记录。
7. 点 "回滚"，正文变回上一次"已应用"或最早版本，历史记录相应变化。
8. "查看历史" 列出所有应用，可回滚到任意历史版本。
9. 加料工作台右侧 "清空加料" 同时把已应用的章节正文还原为最早的原版。
10. 导出 TXT 文件以 `chapters.content` 为准（即已应用过的版本）。

### 12.8 已知限制

- diff 按"句末标点"切分，**单句过长时对齐粒度变粗**；v0.3 计划改用按字符级 LCS 或 Myers diff。
- 整章级应用，无法只应用部分；如需部分应用可手动编辑章节（已支持编辑模式）。
- 多个 suggestion 并存时只能"应用到最新"或"回滚到最近 superseded"，暂不支持"同时保留多版本并切换"。
